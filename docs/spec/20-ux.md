## 1) Navigation & Information Architecture

Two top-level concepts:

- **SSH Profiles**: host connection targets and authentication material.
- **Sessions (Conversations)**: chat-like timelines bound to a profile.

Recommended screens:

1. Profiles List
2. Profile Editor (create/edit)
3. Fingerprint Confirm (first use)
4. Session (chat timeline)
5. Pinned Commands (optional early)
6. Settings (optional early)

## 2) Session Timeline: Message Types

Message types in the chat timeline:

- **Command message** (user-authored)
  - displays with `$ ` prefix
  - actions: edit, copy, rerun, pin
- **Output message** (tool/system)
  - displays stdout and stderr as separate blocks
  - actions: copy, collapse/expand, search, export
- **Status message** (system)
  - connecting / connected / disconnected
  - host key verification prompts

## 3) Output Rendering Rules

Default rendering:

- stdout as a fenced code block using `text`
- stderr as a fenced code block using `stderr` (or `text` + label)

Large output behavior:

- If output exceeds a threshold:
  - show the last N lines/chars
  - show a "truncated" badge
  - provide "copy all" or "export" (with warning)

Streaming behavior:

- Output message starts as empty and appends chunks.
- UI updates are throttled (e.g. 50–100ms batching).

## 4) Host Key Verification UX

First connection:

- show hostname + port + username
- show fingerprint (SHA256)
- actions: "Trust and Continue" / "Cancel"

Fingerprint change:

- default: block connection
- show the previous fingerprint and the new fingerprint
- actions: "Cancel" / "Replace fingerprint (dangerous)"
  - require a second confirmation

## 5) Error UX

Present errors as actionable, non-technical where possible:

- Authentication failed:
  - show key used and whether passphrase was provided
- Network error:
  - show reconnect button
- Host key mismatch:
  - show the security warning and block by default
- Command error:
  - show exit code if available

## 6) Design Direction

We aim for a calm, readable, “chat-first” interface:

- strong typography hierarchy for outputs
- subtle separators, clear affordances for copy/rerun
- minimal chrome; let output be the hero

Important: the implementation must be original (no code copying), but we can match the *feel* of RikkaHub’s chat rendering quality.
